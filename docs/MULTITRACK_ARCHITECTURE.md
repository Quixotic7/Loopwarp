# Elasticat Multitrack Architecture

This document is the target design for growing Elasticat from a single-track sample
player into an 8-track instrument, and the phased roadmap for building it. It
complements `docs/ARCHITECTURE.md`, which covers code organization; this document
covers the feature/DSP design.

## Design decisions

Three foundational choices, confirmed before this design was written:

- **Mod routing is a flexible matrix**, assigned via grid interaction (see below), not
  hardcoded source-to-destination pairs. Capped at 4 destinations per modulator per
  track to keep engine cost bounded.
- **The `neighbor` machine is Elektron-style** (as on Digitakt/Analog Rytm): track N's
  own generator is disabled entirely; track N-1's post-chain audio replaces track N's
  chain input, and track N-1 stops feeding master/sends on its own. Track N then
  applies its own filter/amp/insert FX/routing on top before it reaches master/sends.
- **Track count is configurable from day one**, via an active-track-count setting, so
  the engine only instantiates chains for active tracks. This is a direct response to
  the norns CPU ceiling: dropping the count later is changing a number, not a
  redesign.

## Track model

8 tracks. Per-track scope: **TRIG, SOURCE, FILTER, AMP, FX, MOD** categories (each
track has its own machine/warp choice, note/pitch identity, filter, amp envelope,
insert FX, and mod section). Global scope: **MASTER** (bpm/volume) and **PATTERN**
(length/scale) stay shared across all tracks, plus the 2 Send FX busses, the Master
FX, and the scene/crossfader system.

Norns params for per-track categories are generated by one template function called
once per track index, producing IDs like `elasticat_track3_filter_cutoff` — never 8
hand-written copies of the same param block. `FX` and `AMP` categories mix per-track
pages (insert FX, track amp) with a couple of global pages (Send1/Send2 FX, Master
AMP); the page model needs a way to flag a page as global vs. per-track so the page
resolver can tell the two apart.

The sample pool stays global/per-project (unchanged): all tracks pull from the same
128-slot pool. The sample-slot range grows from 1-128 to 0-128, where 0 means
OFF/no sample (UI shows "OFF"; engine treats slot 0 as silent, no buffer bound).

## Grid layout additions

- **Row 4, first 3 keys**: mod-source select (LFO1 / LFO2 / ENV). Hold one and adjust
  any parameter (on any page, any track) to set/adjust that modulator's depth for that
  parameter on the current track — the exact same interaction shape as step
  parameter-locking (hold a special key, then turn an encoder), just with a different
  held-key context. Bipolar depth, signed by encoder direction. Capped at 4
  destinations per modulator per track.
- **Row 4, columns 8-15**: 8 track-select keys.
- **FN + track key**: mute that track. Muted tracks stay silent regardless of the
  crossfader (see below).
- **Row 5, 8 keys below the track row**: the scene crossfader (Octatrack-style). Key 1
  = scene A anchor, key 8 = scene B anchor, keys 2-7 = crossfade control surface.
  - Hold scene A or B: adjusting any parameter locks its value to that scene — again,
    the same hold-a-key-then-adjust shape as step locks and mod-depth assignment.
    Holding a scene key and pressing a track key toggles whether that track's volume
    is included in the scene morph (Octatrack calls this "scene mute assignment").
  - Hold any of the middle 6 keys and turn E3: live-adjusts crossfade position. Shows
    a popup with the crossfader position, and lights the 6 grid keys proportionally
    (crossfade position downscaled to 6 steps).
  - Moving the crossfader interpolates every scene-locked parameter, and the volume of
    every scene-included track, between their scene A and scene B values.
- **FN + MOD category key**: opens a per-track mod-matrix view — one sub-page per
  modulator (LFO1/LFO2/ENV), listing that modulator's current destinations and
  depths. This reuses the existing FN+category settings-page mechanism
  (`open_param_settings` in `lib/pages/navigation.lua`), just with mod-matrix content
  instead of a flat settings list.

**Shared architectural thread**: step param-locks, mod-depth assignment, and
scene-locks are the same interaction shape — hold a special grid key, then adjust a
parameter, and what that adjustment *means* depends on what's held. `grid_sequencer.lua`
should grow one generalized "held modifier" concept these three features share (Phase
3/4), rather than three separate bespoke code paths bolted on next to each other.

## Engine (SuperCollider) architecture

Per active track, the signal chain is:

```
Machine synth → Filter synth → Amp envelope → Insert FX synth → Master bus
                                                               → Send1 bus (pre/post insert FX, per-track setting)
                                                               → Send2 bus (pre/post insert FX, per-track setting)
```

- **Machine**: the existing tape/tempo_varispeed/chopped/granular/random_ola/
  pitch_corrected readers, unchanged — or, for tracks 2-8, `neighbor` (see below).
- **Filter**: analog LP, morphing LP/BP/HP, or comb. One SynthDef per filter type,
  swapped via the crossfaded synth-replacement pattern already used for warp-mode
  switching ("Added immediate crossfaded mode switching," per `CHANGELOG.md`) — reuse
  that mechanism rather than inventing a new one.
- **Amp envelope**: standard attack/hold/release gate, retriggered per step (this
  already exists per-track in spirit via the current slice envelope params; it becomes
  a first-class per-track stage).
- **Insert FX**: one selectable effect per track, same SynthDef-swap pattern as
  filters.
- **Sends**: 2 global send busses plus a Master FX bus. Each track has a per-track
  setting for whether its send taps happen pre- or post-insert-FX.
- **Mod matrix**: 2 LFOs, 1 envelope, and a drawable/etched custom waveform (used as
  an LFO shape) per track. Each assigned (modulator, destination) pair sums
  `source.kr * depth` into the destination's base value at control rate. The 4-per-
  modulator cap keeps this bounded to a small, fixed-size per-track structure rather
  than an open-ended matrix.

### Neighbor routing

Each track's chain has two routing points that are just bus arguments on already-
running synths, not a synth graph rebuild:

- **Chain input**: normally "this track's own Machine output." When set to `neighbor`,
  it becomes "track (N-1)'s post-chain output bus" instead.
- **Output destination**: normally "Master bus + whichever Sends are enabled." When
  track N is `neighbor`, track (N-1)'s output destination becomes "nowhere" (it no
  longer reaches master/sends on its own — its audio only continues via the bus
  track N is now reading from).

Both are plain bus `.set()` calls on running synths, matching how bus routing already
works elsewhere in `Engine_Elasticat.sc`.

### Active track count

An engine command sets how many tracks are actually instantiated and running. Tracks
beyond that count allocate no synths and no busses. This is the load-bearing answer to
the CPU-ceiling concern: the same engine code scales from 1 to 8 tracks by changing
one number, not by branching on track count anywhere in the DSP.

## Projects

A project bundles:

1. **All norns params** — likely via the existing pset mechanism (`pset_path`/
   `load_startup_pset` in `elasticat.lua`), since per-track params are already
   ordinary norns params once Phase 1 lands.
2. **Sequencer/scene state** — per-track patterns, step locks, mod-depth assignments,
   and scene locks are plain Lua tables today, not norns params, so they need their
   own serialized blob. Follow the existing `tab.save`/`tab.load`-to-`_path.data`
   pattern already used for `save_sample_pool_state`/`save_browser_folder` in
   `lib/script_state.lua`.

Project browsing can reuse the existing `fileselect`-based browser already used for
loading samples, rather than building a new file-picker UI.

## Phased roadmap

Each phase after 0/1 is a substantial DSP or interaction design in its own right, and
gets its own review cycle: audio/DSP behavior and grid feel can only really be
verified on real hardware, not by reading code, so bundling many phases into one
unreviewable diff would be a mistake regardless of who's implementing it.

**How to use the acceptance criteria below**: every phase's criteria must hold before
starting the next phase. "Regression check" items call out things earlier phases
established that must still work — verify those first, since a regression there
means the new phase's changes broke something rather than that the new phase itself
is incomplete.

### Phase 0 — Modularity refactor *(done)*

Extracted `lib/wav_reader.lua`, `lib/script_state.lua`, `lib/pages/model.lua`,
`lib/pages/navigation.lua`, `lib/ui/param_values.lua`, `lib/ui/source_page.lua` out of
`elasticat.lua`; deleted confirmed dead code; converted `format_item_value`'s
id-keyed if/elseif chain to a lookup table. Pure code motion, no behavior change.

**Acceptance criteria:**
- `bin/test-elasticat-lua`, `bin/test-elasticat-sclang`, and `git diff --check` all pass.
- Every page (MASTER, PATTERN, TRIG, SOURCE incl. all 4 sub-pages, FILTER, AMP, FX,
  MOD, and every settings page) renders identically to before the refactor.
- Every existing interaction still works: encoder edits and snapping, step
  parameter-locks (set and clear, single-step and all-steps), K2/K3 pair memory per
  page, sample browsing/loading, category/page navigation, FN+category settings
  entry/exit.
- No new dead code introduced by the refactor itself.

### Phase 1 — Track scaffolding

Programmatic per-track param generation (`lib/tracks/params_spec.lua`),
`grid_sequencer.lua` made track-aware (an array of per-track pattern state, operating
on whichever track is selected), track-select/mute grid keys, and a minimal engine
change that instantiates N independent copies of the *existing* machine/warp chain
(no filter/amp-env/insert-fx/mod yet) mixed straight to master, gated by the
active-track-count setting. Goal: 8 tracks each independently playing/sequencing
samples, before any new DSP is added.

**Acceptance criteria:**
- Per-track params exist for every track up to the configured count, with ids
  following a consistent `elasticat_trackN_*` pattern; no hand-written per-track
  duplication in the param definitions.
- Selecting a track switches which pattern/step-locks are being edited and played,
  with zero cross-contamination between tracks (editing track 3 never touches track
  5's pattern).
- Track-select keys (row 4, cols 8-15) switch the active track; FN+track mutes it,
  and a muted track produces no audio regardless of what else is happening.
- The engine only instantiates/runs chains for tracks up to `active_track_count`;
  tracks beyond that count consume no synths or busses.
- **Hardware test**: load 8 different samples across 8 tracks, give each a distinct
  pattern/pitch/machine, and confirm all 8 play back correctly and simultaneously
  with no glitches or dropouts. Drop `active_track_count` to a lower number and
  confirm CPU load actually decreases (norns' CPU/load meter, or audible glitching as
  a practical proxy if no meter is available) and the dropped tracks go silent.
- **Regression check**: with track 1 selected and tracks 2-8 at default/inactive,
  behavior matches Phase 0 exactly (this confirms the single-track path wasn't
  broken by making everything track-aware).

### Phase 2 — Filter + Amp envelope + Insert FX chain

The filter types, envelope, and one insert FX type first, more FX types after.
Grid/UI to select and edit them, reusing the Phase 0 UI modules.

**Acceptance criteria:**
- All three filter types (analog LP, morphing LP/BP/HP, comb) exist as separate
  SynthDefs and switch via the existing crossfaded-replacement pattern.
- Amp envelope (attack/hold/release) is audible and adjustable per track.
- At least one insert FX type is implemented and switchable the same way as filters.
- FILTER and (per-track) AMP/FX pages let you select and edit all of the above per
  track, using `lib/ui/param_renderer.lua` / `lib/ui/source_page.lua` conventions.
- **Hardware test**: sweep filter cutoff/resonance while a track plays — no zipper
  noise or clicks. Switch filter types live — clean crossfade, no pop or dropout.
  Change envelope attack/hold/release — audibly distinct shapes. With all 8 tracks
  active and each running a filter + insert FX, confirm no audio dropouts.
- **Regression check**: Phase 1's per-track playback/sequencing still works
  identically with filters/envelope/FX left at neutral/bypass settings.

### Phase 3 — Mod section

2 LFOs + 1 envelope + custom-drawn LFO shape per track, grid-based depth assignment,
mod-matrix view page. The held-modifier generalization in `grid_sequencer.lua` lands
here (shared with Phase 4).

**Acceptance criteria:**
- 2 LFOs, 1 envelope, and a drawable/etched custom waveform exist per track and can
  be used as modulation sources.
- Holding a mod-source key (row 4, first 3 keys) and adjusting any parameter
  sets/adjusts that (modulator, parameter) depth on the current track; the
  4-destinations-per-modulator cap is enforced.
- FN+MOD opens a mod-matrix view with one sub-page per modulator, showing that
  modulator's current destinations and depths for the selected track.
- `grid_sequencer.lua`'s held-modifier concept is generalized (not three separate
  bespoke code paths for step-locks vs. mod-depth vs. scenes).
- **Hardware test**: assign an LFO to filter cutoff and confirm audible periodic
  modulation at the expected rate/depth; assign the envelope to amp and confirm it
  follows triggers; draw a custom waveform and confirm the LFO follows the drawn
  shape.
- **Regression check**: step parameter-locking (Phase 0) still works exactly as
  before now that the held-modifier code is shared/generalized.

### Phase 4 — Scene crossfader

Scene locks, track-inclusion toggling, live crossfade with popup + grid LEDs, morph
interpolation.

**Acceptance criteria:**
- Holding scene A or B and adjusting a parameter locks its value to that scene, for
  parameters on any track.
- Holding a scene key and pressing a track key toggles that track's inclusion in the
  scene morph; muted tracks stay silent regardless of crossfade position.
- Holding any of the middle 6 grid keys and turning E3 live-adjusts the crossfade
  position, shows the popup, and lights the 6 keys proportionally.
- Moving the crossfader smoothly interpolates every scene-locked parameter and every
  included track's volume between scene A and scene B values.
- **Hardware test**: lock a filter cutoff to a low value at scene A and a high value
  at scene B, confirm a smooth audible sweep as the crossfader moves with no zipper
  noise; confirm unlocked parameters and non-included tracks are unaffected by
  crossfading; confirm muted tracks stay silent across the whole crossfade range.
- **Regression check**: mod-depth assignment and step-locks (Phases 0/3) still work
  unchanged now that scenes share the held-modifier mechanism.

### Phase 5 — Neighbor machine + bus rerouting

**Acceptance criteria:**
- Setting track N (2-8) to `neighbor` disables its own generator and routes track
  (N-1)'s post-chain audio into N's filter input via a bus `.set()`, not a synth
  rebuild.
- Once consumed by a neighbor, track (N-1) no longer independently reaches
  master/sends.
- Switching a track in and out of neighbor mode at runtime produces no clicks, pops,
  or hangs.
- **Hardware test**: chain 2-3 consecutive tracks via neighbor and confirm the
  combined signal only exits via the last track in the chain; confirm filter/FX
  changes on the neighbor-consuming track audibly affect the whole combined signal.
- **Regression check**: non-neighbor tracks (including ones adjacent to a neighbor
  chain) are unaffected.

### Phase 6 — Send FX (2 global) + Master FX + pre/post-insert send tap setting

The extra FX-category and AMP-category global pages.

**Acceptance criteria:**
- 2 global send busses and a master FX bus exist and are reachable from every track.
- The per-track pre/post-insert-FX send tap setting produces an audibly different
  signal (dry pre-insert vs. processed post-insert) when toggled.
- The FX category shows the per-track insert FX page plus Send1/Send2 FX as
  additional global pages; the AMP category shows per-track amp plus Master AMP as
  an additional global page (using the `page.global` flag from `lib/pages/model.lua`).
- **Hardware test**: send a track to a send FX both pre- and post-insert, confirm the
  audible difference matches expectations; confirm send levels sum correctly at
  master with no clipping surprises at default gain staging.
- **Regression check**: insert FX (Phase 2) and neighbor routing (Phase 5) still
  behave correctly with sends now also active.

### Phase 7 — Projects

Save/load all params + sequencer/scene state, project browser.

**Acceptance criteria:**
- Saving a project captures all norns params plus the full per-track sequencer state
  (patterns, step locks, mod-depth assignments, scene locks).
- Loading a project restores an exact prior session's behavior.
- The project browser works via the existing `fileselect` pattern already used for
  samples.
- **Hardware test**: build a session with distinct per-track patterns, locks, scenes,
  and mod assignments; save; power-cycle or reload the script entirely; load the
  project; confirm behavior matches the pre-save session exactly.
- **Regression check**: loading an *old* pset (pre-projects) doesn't crash — define
  and document the expected fallback behavior.

### Phase 8 — Performance pass

Once the above is playable, profile actual CPU headroom on hardware and tune the
active-track-count ceiling, mod-matrix cost, and filter choices accordingly.

**Acceptance criteria:**
- CPU usage is measured at each active-track-count setting (1 through 8) with full
  chains (filter + amp + insert FX + mod) running.
- A practical maximum track count for reliable, glitch-free playback on real
  hardware is determined and written down (in this doc or `CHANGELOG.md`).
- Any optimizations applied are verified with an A/B comparison to confirm no
  audible behavior changed, only performance.
